# 制作方針

## 収集対象
- 速報・話題は、原則として「JSTにおいて前日に発生した・確認された出来事」を中心に収集する。
  - **配信日**（dateLong の日付）= 新聞を届ける日（JST当日）
  - **記事が扱う出来事の日**（ファイル名の YYYY-MM-DD）= 配信日の前日（JST昨日）
- 直近 7 日以内に同一内容を記録している場合は、内容が更新されない限り追加しない。
- 既に出ている話題を別の言い回しで繰り返すのではなく、実際に新しい事実がある場合だけ追記する。

## 未来情報の保留スタック（`stock/`）

公式は「来月のライブ」「数ヶ月先の総選挙結果」のように**先の予定をまとめて発表**する。これを発表日に1本で書き切ると、肝心の**開催前日・当日の「明日開幕／本日開幕」記事**や**閉幕レポ**を書くときに手元に情報がなく、毎回ゼロから検索し直すことになる（検索で出てこない場合もある）。そこで「いま判明した未来の確定事実」を**いったん `stock/` に寝かせ**、配信日が近づいたら取り出して記事化する。

- 場所: リポジトリ直下の **`stock/`**。`docs/` の**外**なので **Jekyll では公開されない**が、git 管理のため**実行環境を作り直しても永続化**される。
- 中身: `stock/upcoming.yml`（未来イベントのインデックス、`event_start` 昇順）と `stock/notes/`（長文メモ）。
- **詳細スキーマ・運用フローは `stock/README.md` を必ず参照する。**

### 保留する（情報を入れる）
未来の予定を見つけたら（または既存記事で先の予定に触れたら）、`stock/upcoming.yml` にエントリを追加する。`kind`（公式/ファン）、`status: pending`、`confidence`（confirmed/unconfirmed）、`source` を明記し、`facts` には記事素材になる粒度で確定事実を書く。`publish_plan` に「いつ（配信日）・どの切り口で」記事化するかを記す。

**非公式・ファン企画枠（`kind: ファン`）**：応援広告・誕生日センター広告・有志オフ・記念日など、ファン主導の盛り上がりも対象。掲出/開催が先に告知されるので「前日に予告→当日に本日掲出」と continuity 記事化できる。`source` は**主催者自身の公開告知**を必ず置く。**リーク（非公開情報の暴露）は扱わない。** 人を傷つける内容は記録しない。

### 取り出す（毎日の記事生成時・必須）
1. 配信日（JST当日）を取得する。
2. `stock/upcoming.yml` を走査し、**`publish_plan[].on` が配信日と一致**／**`event_start` が配信日〜翌日以内**／**`event_end` の翌日**に該当するエントリを記事化候補にする。
3. 該当エントリの `facts` を素材に記事を書く（可能なら Web で追検証）。「明日開幕→本日開幕→昨日閉幕（レポ）」と**同一イベントを複数日に渡って continuity 記事化**できるのがこの仕組みの狙い。
4. 記事化したら `used_in` に slug を追記し、`status` を更新（続きあり→`partially-used`／消化済み→`done`）。生配信などで新情報が出たら関連エントリの `facts` を更新する。

### 日付モデル（配信日ルールと一致）
出来事の日付を `E` とすると、「本日◯◯」記事は配信日 `E`、「明日◯◯」記事は配信日 `E-1`、「昨日◯◯（レポ）」記事は配信日 `E+1`。よって `publish_plan` の `on:` には**配信日（edition_date）**を書く。

## 日付・時制のルール（配信日基準）

記事の日付表現はすべて**配信日（dateLong の日付＝JST当日）を基準**にして書く。
記事を取材・執筆したのは前日だが、出力される紙面は当日である。その当日読者の視点で相対表現を決める。

### 相対表現の基準

| 相対表現 | 意味 | 例（配信日6/6の場合） |
|----------|------|----------------------|
| 昨日・きのう | 配信日の前日 = 記事対象日 | 6/5の出来事・6/5に発表されたこと → 「昨日確認」✅ |
| 本日・今日 | 配信日当日 | 6/6開幕 → 「本日開幕」✅ |
| 明日・あした | 配信日の翌日 | 6/7の予定 → 「明日開催」✅ |

**❌ やってはいけないパターン：**  
前日（6/5）の書き手視点で「明日開幕（6/6）」と書いたまま 6/6 の紙面に載せる。  
読者には「明日＝6/7」と読めてしまう。必ず配信日基準に直す。

### badge フィールドの記法

- 配信日に開始するイベント → `"本日開始"` または `"開幕"`
- 配信日より前に始まっていて継続中 → `"開催中"` / `"受付中"`
- 配信日翌日以降に始まる → `"MM/DD開始"` など絶対日付
- `"新発表"` / `"更新あり"` など時制に依存しない表現は常に使用可

### editions.yml の indexNote ルール

- `indexNote` は**前日（記事対象日）**に確認された出来事を列挙する
- 配信日当日に開幕するイベントは「本日開幕」と書いてよい（配信日基準で正しい）
- ❌ 禁止：前日の書き手視点での「明日開幕」をそのまま貼り付ける

### 確認チェック

記事生成後、以下を必ず確認する：
1. `date -u -d '+9 hours' +%Y-%m-%d` でJST当日日付（配信日）を取得する
2. **配信日** = JST当日、**記事対象日** = JST昨日 であることを確認する
3. ファイル名の日付が「JST昨日」であることを確認する
4. title・lead・本文の「昨日/本日/明日」がすべて**配信日基準**になっているか確認する
5. 前日の書き手視点で書かれた相対表現が残っていないか確認する

## 記録ルール
- すでに掲載した内容の再録、要約の重複、同一ニュースの別ページ転載は避ける。
- 追記がある場合は「更新あり」と明示し、差分だけを記録する。

## 生成時の確認
- 記事の見出し・本文・メタ情報が、直近 1 週間の重複を含まないかを必ず確認する。
- もし同じ話題が複数回出ている場合は、最新の更新情報のみ残す。
- 出来る限り網羅的に、かつ重複のないように記録することを心がける。
- 1日の記事は最低10件以上を目指す。最大は100件とする。
    - 公式情報は網羅的にカバーする
    - 非公式情報も、盛り上がっている場合は積極的に記録する
        - 特に人を傷つける話題でなければ積極的に記録する。
- 書き出しは常に1記事ずつ行う。

## ファイル管理
- 記事ファイルは `docs/_posts/YYYY/MM/` のように年月サブディレクトリに格納する。
  例: `docs/_posts/2026/06/2026-06-05-cg-event.md`
- ファイル名の形式: `YYYY-MM-DD-BRAND-EVENT[-YYYYMMDD].md`
  例: `2026-06-05-cg-sousenkyo-20260605.md`
  複数日で継続する話題には日付サフィックスを付け、別記事として管理する。
- `edition` フィールドは廃止済み。frontmatter に書かない。
- **`edition_date`（配信日）** を必ず frontmatter に記載する。
  - ファイル名の YYYY-MM-DD（記事対象日）の翌日 = 配信日
  - 例: ファイル `2026-06-05-*.md` → `edition_date: "2026-06-06"`
  - front.html・brand.html はこのフィールドで記事を号ごとに集約する

## editions.yml の更新ルール
- editions.yml は**リスト形式**（最新号が先頭）。号を更新する際は先頭に新しいエントリを追加する。過去の号は削除しない（バックナンバー棚に使用）。
- `date` は配信日（JST当日）。`date -u -d '+9 hours' +%Y-%m-%d` で確認。
- `dateLong` は配信日の曜日付き長形式。
- `pages` は「全 NN 面」（当日の記事数に応じて）。
- `stamp` は「本日 06:00 締切」固定。
- 号数は `masthead.html` が `founding_date` と `editions[0].date` から自動計算するため、editions.yml には書かない。
- `indexNote` は配信日基準で書く（配信日に開幕するイベントは「本日開幕」と書いてよい）。

## 毎日の更新作業チェックリスト
0. **`stock/upcoming.yml` を確認**し、配信日に該当する予定（`publish_plan[].on` 一致／`event_start` 直近／`event_end` 翌日）を記事化候補に入れる。記事化したら `used_in`・`status` を更新（詳細は「未来情報の保留スタック」節）。
1. `docs/_posts/YYYY/MM/` に記事ファイルを作成（10件以上目安）
   - frontmatter に `edition_date: "YYYY-MM-DD"` を記載（ファイル名日付の翌日 = 配信日）
   - 新たに判明した**未来の予定**は `stock/upcoming.yml` に保留する
2. `docs/_data/editions.yml` を更新
   - リスト先頭に新号エントリを追加（date・dateLong・pages・stamp・indexValue・indexNote）
   - 過去のエントリは削除しない
3. `docs/_data/ranking.yml` を更新（8件のランキング）
4. `docs/_data/editorial.yml` を更新（社説タイトル・本文）
5. `python3 scripts/update_birthdays.py` を実行（birthdays.yml 自動更新）
6. コミット＆プッシュ

## 誕生日スクリプト
`scripts/update_birthdays.py` を実行すると `docs/_data/birthdays.yml` が自動更新される。
Claude Code が実行責務を持つ（GitHub Actions は未設定）。
