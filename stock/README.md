# stock/ — 未来情報の保留スタック（編集部の取材ノート）

このディレクトリは **記事化前の「未来情報」を寝かせておく非公開の取材ノート**です。

- `docs/` の **外**にあるため Jekyll でビルドされず、**サイトには公開されない**。
- ただし git で管理されるため、**実行環境が作り直されても情報は永続化**される。
- Claude Code（編集部）が毎日の記事生成時に**まず参照し、配信日に該当する予定を記事化**する。

## なぜ必要か

公式は「7月のライブ」「9月の総選挙結果」のように**先の予定**をまとめて発表する。
これをその日に1本の記事にして終わらせると、肝心の**開催当日／前日の「本日開幕」「明日開幕」記事**を
書くときに情報が手元になく、毎回ゼロから Web 検索し直すことになる（しかも検索で出てこないこともある）。

そこで「いま分かった未来の確定事実」をこのスタックに**いったん保留**しておき、
配信日が近づいたタイミングで**保留分を取り出して記事化**する。これにより：

- 「明日開幕」→「本日開幕」→「昨日閉幕（レポ）」と**同じイベントを複数日に渡って continuity 記事化**できる。
- 当日の検索失敗に依存せず、**確定事実をストックから引いて**安定して紙面を作れる。

## ファイル構成

```
stock/
├── README.md      # このファイル（運用説明・スキーマ定義）
├── upcoming.yml   # 未来イベント・予定情報のインデックス（event_start 昇順）
└── notes/         # 任意：1イベントの長文取材メモ（upcoming.yml に収まらない詳細）
```

## upcoming.yml のスキーマ

最小単位は「1つの未来の出来事（イベント／発売／配信／投票フェーズ等）」。
`event_start` の昇順で並べる（直近が先頭）。

```yaml
- id: iwsf2026                 # 一意ID（slug 系。記事 slug の接頭辞に流用可）
  title: "IDOL WORLD SUPER FESTIVAL 2026"
  brand: godo                  # 記事化時の brand 候補（sogo/as765/cg/ml/sc/sidem/gaku/valiv/godo/other）
  kind: 公式                    # 公式 / ファン（記事 frontmatter の kind と対応）
  event_start: "2026-07-24"    # 出来事が起こる日（複数日なら開始日）。JST。
  event_end:   "2026-07-26"    # 終了日（単日なら省略可）
  status: pending              # pending / watching / partially-used / done / dropped
  confidence: confirmed        # confirmed（当事者の公開告知あり）/ unconfirmed（要追検証）。※リークは扱わない
  added: "2026-06-06"          # スタックに保留した配信日
  source: "情報源（URL や媒体名。ファン企画なら主催者の公開告知）"
  facts: |                     # 確定事実。そのまま記事の素材に使える粒度で。複数行可。
    会場: ...
    開場/開演: ...
    チケット: ...
  publish_plan:                # いつ・どの切り口で記事化するか（on は「配信日 edition_date」基準）
    - on: "2026-07-23"  angle: "明日開幕予告"          # 配信日=開催前日
    - on: "2026-07-24"  angle: "本日開幕／DAY1"        # 配信日=開催当日
    - on: "2026-07-27"  angle: "閉幕レポ（昨日千秋楽）" # 配信日=開催翌日
  used_in: []                  # 既に記事化した記事の slug を追記していく
  notes: "補足・未確定点・要追加取材など"
```

### 日付モデル（重要・CLAUDE.md の配信日ルールと一致）

- **配信日（edition_date）= JST当日**、**記事ファイル名の日付 = 配信日の前日**。
- ある出来事の日付を `E` とすると、記事は次のように作る：
  - 「**本日**開幕」予告記事 … 配信日 `E`（ファイル名 `E-1`）
  - 「**明日**開幕」予告記事 … 配信日 `E-1`（ファイル名 `E-2`）
  - 「**昨日**開催（レポ）」記事 … 配信日 `E+1`（ファイル名 `E`）
- したがって `publish_plan` の `on:` には **配信日（edition_date）** を書く。

## 非公式・ファン企画枠（`kind: ファン`）

公式の予定だけでなく、**ファン主導の盛り上がり**もこのスタックの主役。例：

- **応援広告／誕生日センター広告**：「〇〇駅に△△の誕生日広告を出します」（掲出開始日が先に告知される）
- **痛車オフ・有志同人イベント・合同誌即売**：開催日が先に告知される
- **記念日・節目**：周年、生誕祭、楽曲N周年 など（CLAUDE.md の誕生日スクリプトと別軸の「予定」）

これらは「掲出開始／開催の**前日に予告**、当日に『本日掲出』」という形で公式イベントと同じく continuity 記事化できる。だからこそ事前にスタックへ保留する価値が大きい。

**ルール：**
- `kind: ファン`、`brand` は内容に応じて（横断的なら `other`）。
- `source` は**主催者自身の公開告知**（公開アカウントの募集ツイート・特設ページ等）を必ず置く。当事者が公開していないものは載せない。
- `confidence: confirmed` は「当事者が公に告知済み」。実施が流動的なら `unconfirmed` にして当日までに追検証。
- **リーク（非公開情報の暴露）は扱わない。** 未発表のネタバレ・内部情報は保留しない。
- 人を傷つける内容は記録しない（CLAUDE.md の収集方針に準拠）。

## 運用フロー

### 1. 保留する（情報を入れる）
新しい未来情報を見つけたら（または既存記事で先の予定に触れたら）、
`upcoming.yml` にエントリを追加する。`status: pending`、`confidence` を明記。
長い詳細は `notes/<id>.md` に書き、`upcoming.yml` からは要点だけ持つ。

### 2. 取り出す（毎日の記事生成時）
1. `date -u -d '+9 hours' +%Y-%m-%d` で **配信日（JST当日）** を取得。
2. `upcoming.yml` を走査し、次のいずれかに該当するエントリを記事化候補にする：
   - `publish_plan[].on` が **配信日と一致**する
   - `event_start` が **配信日〜翌日以内**（予告・本日記事の書き時）
   - `event_start` <= **配信日** <= `event_end`（イベント開催中・中日記事の書き時）
   - `event_end` の **翌日**（閉幕レポの書き時）
3. 該当エントリの `facts` を素材に記事を書く。可能なら最新情報を Web で**追検証**する。
4. 記事化したら `used_in` に slug を追記し、`status` を更新
   （まだ続きがある→`partially-used` / すべて消化→`done`）。

### 3. 掃除する
`status: done` かつ `event_end` が**過去**のエントリは、四半期ごとを目安に削除してよい
（記事自体は `docs/_posts/` に残るのでスタックから消えても困らない）。
`dropped`（中止・流れた）も同様に整理する。
